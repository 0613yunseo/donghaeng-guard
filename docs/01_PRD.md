## 1. 핵심 한 문장

**동행가드는 전동 휠체어 이용자가 이동 중 마주치는 장애물·단차·급경사 등 노면 위험을 센서로 실시간 감지하고, 앱을 통해 즉각적인 경고와 위치 기반 위험 기록을 제공하는 이동 안전 보조 서비스다.**

---

## 2. 서비스 개요

동행가드는 전동 휠체어에 부착된 **ESP32 기반 센서 모듈**과 모바일 앱을 연동하여, 주행 중 발생하는 위험 요소를 실시간으로 감지하고 사용자에게 경고하는 서비스이다.

센서 모듈은 ToF 센서와 초음파 센서를 통해 장애물, 단차, 근거리 위험 요소를 감지한다. ESP32는 센서값을 1차 처리하고, BLE를 통해 앱으로 위험 상태를 전달한다. 앱은 사용자가 위험을 즉시 인지할 수 있도록 SAFE / WARNING / DANGER 상태를 표시하고, 위험 이벤트 발생 시 GPS 위치와 함께 백엔드 서버로 데이터를 전송한다.

백엔드는 주행 세션과 위험 이벤트를 저장하고, AI 분석 모듈은 누적 데이터를 기반으로 위험 구간을 점수화하거나 향후 안전 경로 추천에 활용한다. 프로젝트 문서에서도 동행가드는 센서 모듈, 모바일 앱, 백엔드 서버, AI 분석 모듈로 구성되며, ESP32가 위험 요소를 1차 판단 후 BLE로 앱에 전달하는 구조로 정리되어 있다.

---

## 3. 문제 정의

전동 휠체어 이용자는 이동 중 보도 턱, 단차, 장애물, 급경사, 불법 주정차 차량 등 다양한 위험 요소에 노출된다. 그러나 일반 지도 서비스나 길 안내 앱은 보행자 중심으로 설계되어 있어, 전동 휠체어 이용자가 실제 주행 중 마주치는 노면 수준의 돌발 위험을 실시간으로 감지하거나 경고하지 못한다.

기존 접근성 지도 서비스는 사전에 구축된 접근 가능 장소 정보나 경로 정보를 제공하는 데 초점이 맞춰져 있다. 따라서 실제 이동 중 갑자기 발생하는 장애물, 단차, 공사 구조물 같은 위험을 즉시 반영하기 어렵다.

동행가드는 이 문제를 해결하기 위해 **센서 기반 실시간 위험 감지 → 앱 알림 → 서버 저장 → AI 분석 → 위험 구간 확인**으로 이어지는 구조를 제공한다.

---

## 4. 앱의 역할

동행가드 앱은 단순 화면이 아니라 **센서 데이터 허브 + 사용자 경고 인터페이스 + 백엔드 전송 중간 지점**이다.

앱은 다음 역할을 수행한다.

```
1. ESP32 센서 모듈과 BLE로 연결한다.
2. 센서에서 거리값과 위험 상태를 실시간으로 수신한다.
3. SAFE / WARNING / DANGER 상태를 화면에 표시한다.
4. 위험 상황 발생 시 사용자에게 즉각적인 알림을 제공한다.
5. GPS 위치를 수집하여 위험 이벤트와 함께 백엔드로 전송한다.
6. 주행 시작/종료를 통해 이동 세션을 관리한다.
7. 주행 기록과 위험 발생 위치를 조회한다.
8. AI 분석 결과 기반 위험 구간 또는 안전 경로 결과를 표시한다.

```

앱 요구사항 문서에서도 앱은 BLE 연결 관리, 위험 이벤트 수신, 위험 알림 제공, GPS 위치 수집, 주행 세션 기록, 위험 이벤트 기록, 서버 전송, 안전 경로 요청/표시를 담당하는 것으로 정리되어 있다.

## 9.2 앱 기술스택 추천

앱은 BLE, GPS, 백그라운드 알림, 권한 처리가 중요하므로 아래 두 가지를 우선 검토한다.


| 후보                    | 장점                   | 주의점            |
| --------------------- | -------------------- | -------------- |
| Android Native Kotlin | BLE/GPS/백그라운드 제어 안정적 | Android 전용 개발  |
| React Native CLI      | UI 개발 속도 빠름, 확장성 좋음  | BLE 네이티브 설정 필요 |


**MVP 안정성 기준으로는 Android Native Kotlin을 우선 추천**한다.

다만 팀의 React 경험이 많고 BLE 라이브러리 설정이 가능하다면 React Native CLI도 가능하다.

---

## 9.3 백엔드/AI 기술스택 추천

백엔드는 사용자, 디바이스, 주행 세션, 위험 이벤트를 안정적으로 관리해야 하므로 Spring Boot 또는 FastAPI 중 선택한다.


| 파트        | 추천                                  |
| --------- | ----------------------------------- |
| 백엔드       | Spring Boot                         |
| AI 분석 API | FastAPI                             |
| DB        | MySQL 또는 PostgreSQL                 |
| 통신        | REST API                            |
| 인증        | MVP에서는 간편 userId/deviceId, 확장 시 JWT |


AI는 초기에는 복잡한 모델보다 **위험 이벤트 누적 횟수, 위험 단계, 좌표 밀집도 기반 규칙 분석**으로 시작하고, 데이터가 쌓이면 모델 기반 분석으로 확장한다.

## 10. 시스템 연동 구조

```
[ESP32 센서 모듈]
ToF / 초음파 거리 측정
SAFE / WARNING / DANGER 판단
BLE 데이터 송신
        ↓
[모바일 앱]
```

## 12. ERD 반영 방향 요약

ERD 상세 문서는 백엔드 파트에서 별도 작성한다.

PRD에서는 앱 요구사항 기준으로 필요한 테이블 방향만 정의한다.


| 테이블         | 필요 이유             |
| ----------- | ----------------- |
| users       | 사용자별 주행 기록 관리     |
| devices     | 센서 모듈 식별 및 연결 관리  |
| trips       | 주행 시작/종료 세션 관리    |
| risk_events | 위험 감지 이벤트 저장      |
| risk_zones  | AI 분석 기반 위험 구간 저장 |


후순위 확장 테이블은 다음과 같다.

```
route_recommendations
notifications
device_logs
user_settings

```

## 13. 기능/비기능 요구사항 분리 기준

기능 요구사항과 비기능 요구사항은 별도 문서로 분리한다.

PRD에서는 아래 수준까지만 정의한다.


| 구분       | PRD에서 다루는 범위          | 별도 문서에서 다룰 범위              |
| -------- | --------------------- | -------------------------- |
| 기능 요구사항  | 핵심 기능 목록, MVP 우선순위    | 상세 FR ID, 설명, 검증 기준        |
| 비기능 요구사항 | 실시간성, 안정성, 보안, 사용성 방향 | NFR ID, 수치 기준, 테스트 방식      |
| API      | 핵심 연동 필요성             | Endpoint, Request/Response |
| ERD      | 필요 테이블 요약             | PK/FK, 컬럼, 관계              |
| BLE      | 센서 연동 필요성             | BLE UUID, 데이터 포맷, 전송 주기    |

## 18. 최종 요약

동행가드 앱은 전동 휠체어 이용자의 이동 중 위험을 실시간으로 감지하고, 사용자에게 즉각적인 경고를 제공하는 안전 보조 앱이다.

이 앱의 핵심은 단순한 지도나 기록 기능이 아니라, **센서 데이터 수신 → 위험 상태 표시 → GPS 기반 위험 이벤트 저장 → AI 기반 위험 구간 분석**으로 이어지는 통합 흐름이다.

MVP에서는 우선 **센서 연결, 실시간 위험 상태 표시, 위험 알림, GPS 기반 위험 이벤트 저장**을 구현하고, 이후 주행 기록, 위험 구간 지도, 안전 경로 추천으로 확장한다.
